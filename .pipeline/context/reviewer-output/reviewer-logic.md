## Logic Review: impact-mapped test traceability via @covers tags

### Issues

- [ ] **AC-6 test assertions under-specify the verifier's logic flow** — Task 1 verify step, assertion (a) says "a touched src file with a covering test triggers `node --test <testFile>` and reports PASS/FAIL" but does not specify what the test should assert about the _ordering_ of operations or the _isolation_ of test execution. The test must verify that: (1) the verifier resolves touching-test mapping correctly (parser → map lookup), (2) batched subprocess with multiple test files succeeds or fails atomically per file, (3) stderr reporting of gaps is distinct from stdout test output. Without explicit sub-assertions the test may pass weakly (e.g., asserting only that subprocess spawning occurs, not that file paths are correctly mapped and results aggregated).

- [ ] **AC-3 "no @covers references" edge case lacks definition of the boundary** — The plan states a touched src file with zero `@covers` references should emit a `[covers-gap]` line on stderr, but does not define whether "zero references" means: (a) the file exists in no test's `@covers` declarations, or (b) the file is not declared anywhere in the impact map. These are the same in the happy path (map is complete), but diverge if (1) backfill is incomplete, (2) a test file exists but lacks `@covers` entirely, or (3) a new source file is touched that has never been tested. The plan should clarify whether the verifier scans the filesystem to discover untested source files, or only reports gaps for files listed in both "Files modified" _and_ the impact map. Current Task 6 AC-6 wording ("any touched file with no `@covers` entry") is ambiguous — does "entry" mean "in the map" or "declared anywhere in the codebase"?

- [ ] **Backfill heuristic ("strip `-test` suffix") ambiguity on multi-match and zero-match scenarios** — Task 10 AC-10 says backfill uses the heuristic "strip `-test` suffix and match against existing source paths". The plan does not specify: (1) what happens when a test file like `scripts/covers-map-test.mjs` (after stripping `-test`) produces `scripts/covers-map.mjs`, but multiple source files exist that match the pattern (e.g., `scripts/covers-map.mjs` _and_ a legacy `scripts/covers-map-utils.mjs`)? Does backfill pick the first match, ask the user, or skip? (2) What happens when the heuristic produces zero matches (e.g., a test file `scripts/x-y-z-test.mjs` strips to `scripts/x-y-z.mjs` but no such file exists)? Current AC-10 wording ("prepends `// @covers <inferred-src>`") does not address these failure modes.

- [ ] **`--strict-gaps` flag naming inconsistent with project conventions** — Task 6 (AC-6) introduces a `--strict-gaps` flag to the verifier CLI. The plan does not verify whether this flag naming matches existing FORGE conventions for boolean opts (e.g., other scripts use `--force`, `--dry-run`, `--strict`, no dashes in compound words). Recommend checking `scripts/covers-backfill.mjs` and other scripts to confirm naming aligns with the project's flag style before implementation.

- [ ] **Missing edge case: verifier behavior when a test file itself is modified but does NOT declare `@covers`** — The plan covers "touched src file with zero references" but does not specify what happens when the touched file is a test file (`scripts/foo-test.mjs`) that declares no `@covers` tag. Should the verifier: (a) emit a gap warning because it doesn't cover anything, (b) skip it because test files are not in the "Files modified" that the verifier cares about, or (c) warn about the test file needing a `@covers` declaration? Task 6 AC-6 implies only source files are checked, but the boundary is not explicit in the verify step.

- [ ] **AC-7 handoff integration point under-specifies "a gap does not block the pipeline"** — Task 7 says "a gap does not block the pipeline but adds a `[covers-gap]` section to the handoff for reviewers". The plan does not specify: (1) what does "add a section" mean operationally? Does the skill modify the handoff markdown file to append a section, or does it inject a signal line into stderr that the worker captures? (2) if the verifier exits non-zero due to test failures (not gaps), does the pipeline block or proceed? Task 6 AC-6 says verifier "exits non-zero when any test fails or when `--strict-gaps` flag is passed and gaps exist" — so the verifier's exit code is conditional. Task 7's "does not block" must clarify whether the skill checks `exitCode` before deciding to block.

- [ ] **TDD Wave 3 regression scope includes 4 test suites, but only 3 are new** — Task 11 lists dependencies `7, 8, 9, 10` (not 1, 2, 3) and says "Confirm all three new test files pass and the existing suite remains green". But Wave 3 task lists `['scripts/covers-parser-test.mjs', 'scripts/covers-map-test.mjs', 'scripts/covers-verify-test.mjs']` — that is correct (three new). However, the plan states the verifier test `scripts/covers-verify-test.mjs` depends on Task 3 (write the test), and Task 3 is in Wave 1, not Wave 2. This is correct (Wave 1 failing tests, Wave 2 implementation, Wave 3 regression on full suite). But the Regression section at line 199 says "Depends: 7, 8, 9, 10" — those are all Wave 2 tasks. The correct dependency for Wave 3 should be **no explicit numeric dependencies** (Wave 3 always depends on all prior phases completing). The current dep list suggests Wave 3 starts only after Task 10 (backfill), but backfill is optional/operator-triggered — should Wave 3 block on backfill? Current wording is logically muddy.

### Per-criterion verdicts

**Verifier logic flow:** The handoff specification at Task 6 AC-6 is mostly sound but under-specifies the test assertions for the core flow. The three sub-assertions (a, b, c) are present, but assertion (a) "a touched src file with a covering test triggers `node --test <testFile>` and reports PASS/FAIL" does not prescribe what the test actually asserts about the result — it may pass by simply confirming that a subprocess spawns, not that the parser→map→lookup chain executes correctly or that results are aggregated properly. For Wave 1 (failing tests) this is a content-level issue, not a blocking logic error, but it risks a weak test that passes prematurely in Wave 2.

**Missing-coverage edge case:** The plan defines "zero `@covers` references" but does not clarify the system boundary. If a source file is touched but never appears in any test's `@covers` declaration, the verifier should emit a gap. But the plan does not specify whether the verifier actively scans the source filesystem to discover such files, or only reports gaps for files listed in "Files modified" _and_ the impact map (which may be empty for pre-backfill runs). This is a design choice, not a blocker — the implementer must make it, and Task 6 AC-6 does hint at the answer (verifier "resolves covering tests" — implying lookup, not filesystem scan). But the spec is implicit rather than explicit.

**Backfill heuristic:** Task 10 AC-10 does not specify multi-match or zero-match outcomes. The plan says "strip `-test` suffix and match against existing source paths" but does not handle the case where the match is ambiguous (multiple source files fit) or empty (no source file fits the pattern). This is a _design gap_, not a logic error — the implementer must decide. But the verify step for Task 10 does not test these edge cases, which means Wave 1 (failing tests for backfill) may not exercise them. **Recommendation:** Add test cases in Task 10 backfill test (if one exists) or clarify in Task 6 that backfill is best-effort and non-critical (already stated in Trade-off, but should be in the test assertions).

**`--strict-gaps` flag naming:** The flag is introduced at Task 6 AC-6 but not validated against project conventions. Minimal risk — implementer can check existing scripts before implementation. No blocker.

**Test file edge case:** If a test file itself is modified (e.g., `scripts/foo-test.mjs`), it may not declare `@covers` (because it _is_ a test, not a source file being covered). The verifier should skip test files when reading "Files modified" — only source files matter. Task 6 AC-6 does not explicitly state this, but the wording "touched src file" implies source files only. **Recommendation:** Clarify in Task 6 that verifier filters "Files modified" to source paths only (excludes `*-test.js`, `*-test.mjs`, etc.).

**Task 7 integration:** "a gap does not block the pipeline but adds a `[covers-gap]` section to the handoff for reviewers" — the mechanism is unspecified. Task 7 AC-7 says the skill "logs `[covers] <N> tests resolved, <M> gaps` to stderr" and "adds a `[covers-gap]` section to the handoff for reviewers". These are two different actions: (1) logging, (2) modifying the handoff file. The plan does not specify who modifies the handoff — the verifier subprocess, or the skill after capturing verifier output? If the skill must modify the handoff, the AC-7 task description should state this explicitly (e.g., "appends a `## Covers` section to docs/context/handoff.md with details from verifier stderr").

**Wave 3 dependencies:** Task 11 lists "Depends: 7, 8, 9, 10" which are all Wave 2. This is correct for ordering (Wave 3 starts after Wave 2 completes), but it over-specifies — Wave 3 should depend on the completion of Phase 2 as a whole, not individual tasks. The backfill task (10) is operator-triggered and not required for the verifier to function, so Wave 3 may run before backfill is executed. The current wording may confuse the implementer into thinking backfill must complete before Wave 3 can start. **Recommendation:** Remove numeric dependencies from Wave 3 or change to "all Wave 2 tasks" to clarify the phase boundary.

**AC-1, AC-2, AC-4, AC-5, AC-8:** All sound. No logic issues.

### Verdict

REVISE — The plan is logically sound in structure and TDD pacing, but has several edge-case definitions that are implicit rather than explicit, and one integration point (Task 7 handoff modification) that under-specifies the mechanism. These are safe to address during implementation, but the planner should tighten the specifications to reduce implementation ambiguity.

Issues summary:
1. Task 1 AC-3 test assertions should detail the verifier's parser→map→lookup flow more explicitly (sub-assertions for correctness, not just subprocess spawning).
2. Task 3 AC-3 "no @covers references" should clarify whether gaps are for files in the map, or actively discovered by filesystem scan.
3. Task 10 AC-10 backfill heuristic should specify multi-match and zero-match behavior (or note as implementer's judgment call).
4. Task 6 AC-6 should filter "Files modified" to source paths only, excluding test files.
5. Task 7 AC-7 should specify the mechanism for appending the `[covers-gap]` section to the handoff (verifier subprocess or skill post-processing).
6. Task 11 Wave 3 dependencies should be clarified as "phase completion" not individual task completion, to avoid confusion about backfill blocking Wave 3.
