### Fix 6 — `bin/forge.cmd` tracked in git with absolute user paths

**File:** `.gitignore` + git index

**Status: RESOLVED (forge.cmd) / NEW ISSUE FOUND (forge-mcp-server.cmd)**

`bin/forge.cmd` is correctly listed in `.gitignore` (line 36) and confirmed absent from the git index (`git ls-files bin/forge.cmd` returns empty).

However: `bin/forge-mcp-server.cmd` is NOT in `.gitignore` and IS tracked in git. `git ls-files bin/` shows it as a tracked file. The file on disk and in git history (commit `c147f59`) contains:

```

"C:\Users\cuj\OneDrive - Nemlig.com\Skrivebord\node-v24.14.0-win-x64\node.exe" "C:\Users\cuj\forge-plugin\mcp\server.js" %*

This is the same class of leak as the original `forge.cmd` issue: absolute per-user paths committed to the repo. It was generated by the same hook (`hooks/mcp-deps-install.js` line 171–177) and should be treated identically.

**Impact:** Username and machine-specific filesystem layout leaked to git history.

### NEW-2 — `.pipeline/forge-config.json` and `.pipeline/usage.json` not gitignored (MEDIUM)

**File:** `.gitignore`

Both files are currently untracked (`??` in `git status`). The `.gitignore` only ignores specific `.pipeline/` entries: `run-active.json`, `gate-pending.json`, `review-archive/`, `runs/`. It does NOT ignore:

```

### NEW-3 — `mcp/package.json` uses `^` version ranges for all deps (LOW / INFORMATIONAL)

**File:** `mcp/package.json`

All 8 dependencies use `^` ranges:
```json
"@modelcontextprotocol/sdk": "^1.29.0",
"@xterm/headless": "^6.0.0",
"blessed": "^0.1.81",
"ink": "^5.0.0",
"node-pty": "^1.1.0",
"pngjs": "^7.0.0",
"react": "^18.0.0",
"zod": "^3.25.0"
```

`^` allows any non-breaking upgrade (minor + patch). The lockfile (`mcp/package-lock.json`) pins exact versions for the current install, so `npm ci` (now used when lockfile is present) provides deterministic installs. The `^` ranges only become a risk if the lockfile is deleted and `npm install` runs — which regenerates with the latest matching versions.

`forge-core/package.json` uses `"zod": "^3.25.0"` — same pattern, same assessment.

**Mitigating factor:** Both lockfiles are committed and tracked. `npm ci` is preferred when lockfile exists. Risk is residual (lockfile deletion scenario).

**Recommendation:** No immediate action required; lockfiles cover the runtime risk. If you want defense-in-depth, switch `^` to `~` (patch-only) for production deps.
