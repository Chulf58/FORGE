## Original Findings — Verification

### HIGH-1: Gemini API key in URL query string

**Original finding:** `callGemini()` constructed the URL as `?key=<apikey>`, exposing the key in proxy/CDN/load-balancer logs.

**Claimed fix:** Commit `4fce98b` — API key moved to `x-goog-api-key` header.

**Verification:** CONFIRMED RESOLVED.

`mcp/lib/gemini-adapter.js` line 9 shows:
```js
const GEMINI_BASE = 'https://generativelanguage.googleapis.com/v1beta/models';
```
And the fetch call (lines 80–87) constructs `url = \`${GEMINI_BASE}/${modelId}:generateContent\`` with no query parameters. The API key is sent exclusively as:
```js
'x-goog-api-key': apiKey,
```
No `?key=` in any code path. The file header comment explicitly documents this decision. **Status: RESOLVED.**

---

### HIGH-2: Full responseText thrown in Gemini errors (key exposure on 401)

**Original finding:** Error paths threw raw `responseText` which on a 401 response would echo back the API key (Google's 401 body includes the rejected credential in some configurations).

**Claimed fixes:**
- Commit `4fce98b` — `sanitizeErrorMessage()` added, raw body no longer thrown on non-2xx
- Commit `e93d525` — raw body slice removed from JSON parse error path

**Verification:** CONFIRMED RESOLVED.

The non-2xx error path (line 120) calls `sanitizeErrorMessage(response.status, responseText)` which:
1. Parses the body as JSON and extracts only `error.status` or `error.code` (structured fields, not raw content)
2. Falls through to a fixed message `Gemini API error ${status}` if body is non-JSON or has neither field
3. Never interpolates raw `responseText` into any thrown message

The JSON parse error path (line 127) throws:
```js
throw new Error('Gemini response JSON parse failed (unexpected non-JSON body on successful response)');
```
Fixed message — no `responseText.slice()` or body content. The `e93d525` commit message confirms this specific slice was removed.

**Status: RESOLVED.**

### MEDIUM-1: modelId not validated against catalog before API call

**Original finding:** `forge_call_external` at `mcp/server.js:666` accepted any arbitrary `modelId` string and passed it directly to the provider, allowing calls to unintended or injected model IDs.

**Verification:** STILL OPEN.

Reading `mcp/server.js` lines 675–700, the handler:
1. Validates `providerId` against `config.providers` (provider must exist and be enabled)
2. Resolves and checks the API key
3. Passes `modelId` directly to `callOpenAI()` or `callGemini()` **without checking it against `config.models`**

No catalog lookup occurs. An LLM or attacker with tool access can call any arbitrary model string at a given provider — the request goes through as long as the provider's API key is valid. There is no allowlist check of the form:
```js
const knownModel = (config.models || []).find(m => m.id === modelId && m.providerId === provider.id);
if (!knownModel) return errorResult("Unknown model: " + modelId);
```

The `forge_list_models` tool exposes the catalog, so the information to validate against is available at runtime; it just is not consulted. **Status: STILL OPEN.**

**Verification:** CONFIRMED RESOLVED.

- `git ls-files bin/forge.cmd` returns empty — file is not tracked.
- `.gitignore` contains the entry at line 35:
  ```
  # Windows shim — auto-generated by hooks/mcp-deps-install.js on SessionStart
  # with absolute per-user node paths. Never commit: contains username and machine paths.
  bin/forge.cmd
  ```
- The `bb3d96a` commit diff confirms 15 lines were removed from the tracked `bin/forge.cmd`.

**Status: RESOLVED.**

### LOW-1: npm install vs npm ci in mcp-deps-install.js

**Original finding:** `hooks/mcp-deps-install.js` uses `npm install` rather than `npm ci`, which can modify `package-lock.json` and introduce unintended dependency upgrades (supply chain risk).

**Verification:** STILL OPEN (accepted risk, unchanged).

`hooks/mcp-deps-install.js` line 138:
```js
execSync(npmCmd + ' install --prefix "' + target.dir.replace(/\\/g, '/') + '"', { ... });
```

Still `npm install`, not `npm ci`. No change made. The mitigating factor noted in the original audit remains: both `mcp/package-lock.json` and `packages/forge-core/package-lock.json` are tracked in git (verified via `git ls-files`), so drift is at least detectable on review. However, the install command itself can still mutate the lockfile silently on each SessionStart if dependency ranges resolve differently.

**Status: STILL OPEN (unchanged, mitigated by committed lockfiles).**

### LOW-2: Gemini enabled:true by default without opt-in

**Original finding:** `forge-config.default.json` had Gemini's `enabled` set to `true`, meaning any user who bootstraps the config and has `GEMINI_API_KEY` set would have Gemini calls active without explicit opt-in.

**Verification:** STILL OPEN (unchanged by design).

`forge-config.default.json` line 25: `"enabled": true` for the Gemini provider.

This was not changed. The notes field documents the free-tier intent (`"Free tier; quota is per-model per-day"`), suggesting the design intent is for Gemini to be on by default. No new opt-in mechanism was added. The OpenAI provider correctly defaults to `"enabled": false`. The asymmetry between Gemini (on) and OpenAI (off) is a deliberate design choice, not an oversight, but the LOW finding stands as originally classified since there is no explicit user acknowledgment gate before Gemini calls begin.

**Status: STILL OPEN (design choice, no change).**

## New Issues Found

### NEW HIGH-1: openai-adapter.js throws raw responseText on non-2xx errors

**Severity:** HIGH
**File:** `mcp/lib/openai-adapter.js`, line 93
**Finding:**

```js
if (!response.ok) {
  throw new Error('OpenAI API error ' + response.status + ': ' + responseText);
}
```

The full raw response body (`responseText`) is concatenated into the error message on any non-2xx status. On a 401 Unauthorized response, OpenAI's response body typically contains a JSON object with error details — however the raw body could in edge cases echo content derived from the request (including prompt snippets) or contain sensitive error context. More concretely: the Gemini adapter had the exact same pattern fixed in `4fce98b`, but the same fix was not applied to the OpenAI adapter.

This is the same class of finding as HIGH-2 above. The gemini-adapter.js fix commit `4fce98b` explicitly introduced `sanitizeErrorMessage()` for Gemini but did not apply a parallel treatment to `openai-adapter.js`, which was presumably added or updated in the same session.

**Recommendation:** Apply the same sanitization pattern used in `gemini-adapter.js`. Extract only `error.type`, `error.code`, or `error.message` from the parsed JSON body; fall back to a fixed string on parse failure. Never interpolate raw `responseText` directly into thrown errors.

---

### NEW MEDIUM-1: openai-adapter.js exposes raw body in JSON parse error

**Severity:** MEDIUM
**File:** `mcp/lib/openai-adapter.js`, line 100
**Finding:**

```js
throw new Error('OpenAI response JSON parse failed: ' + responseText.slice(0, 200));
```

The first 200 bytes of the raw response body are included in the JSON parse error message. This is the exact same pattern that `e93d525` fixed in `gemini-adapter.js` (which previously had `responseText.slice(0, 200)` in its JSON parse error path). The fix was not applied to the OpenAI adapter.

On a successful HTTP response (2xx) with non-JSON body, the first 200 bytes of the body would be surfaced. While a 2xx non-JSON response is unusual for the OpenAI Responses API, the pattern is unsafe by class.

**Recommendation:** Replace with a fixed message: `'OpenAI response JSON parse failed (unexpected non-JSON body on successful response)'` — matching the Gemini adapter's corrected form.

## Notes

The Gemini-specific fixes in `4fce98b` and `e93d525` are well-executed: the `sanitizeErrorMessage()` function correctly extracts only structured error fields, the JSON parse error path was cleaned up, and the API key header approach is the right pattern. The gap is that `openai-adapter.js` was not given the same treatment — both the non-2xx error path and the JSON-parse error path still expose raw body content. These should be trivial to fix by mirroring the Gemini adapter's approach.

The modelId validation gap (MEDIUM-1) remains the most structurally meaningful open finding: it allows tool callers to direct API spend at arbitrary model IDs not in the configured catalog.
