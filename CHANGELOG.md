### Fixed
- **Worker MCP server resolves project root to main, not worktree (registry-split fix)** — `mcp/server.js:resolveProjectDir` now performs the same worktree→main `.git` gitdir resolution that `resolveMainProjectDir` previously did exclusively. All MCP tool handlers (`forge_update_run`, `forge_create_run`, `forge_advance_stage`, `forge_get_run`, etc.) automatically get main's project root for run-state operations, regardless of whether the worker spawned with `cwd=<worktree>` and `CLAUDE_PROJECT_DIR=<worktree>`. Without this fix, `createWorktree.js:184-188`'s `copyDirSync(.pipeline/, worktree/.pipeline/)` seeded each worktree with a snapshot of run state, and worker writes went to that snapshot — observed in r-61c6a00a where the plan worker's `forge_update_run` for gate1 wrote gate-pending state to `.worktrees/r-61c6a00a/.pipeline/runs/r-61c6a00a/run.json` while the conductor read the stale main copy. Worktree-local needs (gate-pending.json, reset-pill) are handled by runId-aware lookups and explicit `mcp/lib/worker-paths.js` helpers, not by this resolver. Conductor's MCP server doesn't run in a worktree so the gitdir check is a no-op for it — no conductor behavior change. `resolveMainProjectDir` retained as an explicit alias.

### Added
- **Worker progress logging for silent-death post-mortem (closes TODO `708c056f`)** — `mcp/forge-worker.mjs` per-message loop now emits a `{"type":"forge-worker-progress",...}` JSON line via `writeLog` every 25 messages OR every 60 seconds, whichever first. Tracks `messagesSeen`, `agentsCompleted` (count of `task_notification + status:completed` events), `lastAgentType` (most recent `subagent_type` dispatched via the Agent tool), `contextTokens` (latest assistant message's `input + cache_creation + cache_read` totals — approximates current effective context window), `contextEstimate` (`contextTokens / 200_000`, 3-decimal fraction of haiku context limit), `lastOutputTokens`, and ISO timestamp. Useful for `/forge:resume` to distinguish "hit context wall" from "crashed for other reason" when a worker dies mid-processing. Limitation: only fires while the for-await stream is iterating — true idle (empty stream waiting) is not covered.

Race-free under all spawn timings. The `findWorkerTaskFile` helper at `mcp/server.js:134` is retained (no callers now, but harmless and useful for debugging).

### Added
- **Zombie worker prevention** (closes TODO `b1cdef12`) — workers no longer linger after the conductor marks a run terminal. Two complementary mechanisms: (1) `mcp/forge-worker.mjs` per-message loop now polls `readRunData()` (debounced to ≥500ms) and exits cleanly when run status is `completed`, `failed`, or `discarded`; absent/unreadable `run.json` is fail-open. (2) New helper `mcp/lib/worker-pids.js` exports `sweepStalePids(projectDir)` — probes each `.pipeline/worker-pids/<runId>.json` PID with `process.kill(pid, 0)`, classifies via error code (`ESRCH` = dead → sweep; `EPERM` = alive cross-user → keep; no error = alive). Dead-PID `running` runs are marked `failed` with `failureReason: 'worker process no longer alive (swept)'`, then the PID file is deleted (run.json update durable before delete). 200-entry retention cap. The sweep is wired into `forge_create_run` and `forge_advance_stage` before each collision guard so stale entries don't falsely block legitimate spawns. Test in `mcp/zombie-worker-prevention-test.mjs`.
